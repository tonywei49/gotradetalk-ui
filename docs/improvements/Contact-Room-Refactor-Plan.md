# 联系人与聊天室改造计划

## 1. 目标边界

1. 保留当前“加好友”业务流程，本期不改后端 `matrix_room_id` 必填逻辑。
2. 联系人详情页支持“共同聊天室选择 + 开新聊天室（邀请制）”。
3. 逐步回归 Matrix 原生房间生命周期，减少对单一 DM 房间的硬绑定。
4. 先做低风险前端改造，再做结构性统一。

## 2. 实施计划表

| 编号 | 阶段 | 交付结果 | 关键改动 | 验收标准 | 预计工时 |
|---|---|---|---|---|---|
| 1 | 基线冻结 | 本期范围锁定 | 保持后端 `requestContact` 仍要求 `matrix_room_id`；不改 Hub 表结构 | API 契约不变 | 0.5 天 |
| 2 | 联系人页核心流程 | 可选择共同聊天室并进入 | 联系人详情展示共同聊天室；`聊天` 按钮进入选中房间 | 能稳定进入任意共同房 | 1 天 |
| 3 | 开新聊天室流程 | 创建新房并邀请可用 | `开新聊天室` 执行 `createRoom + invite`，创建后进入房间 | 对方能收到邀请并同意加入 | 0.5-1 天 |
| 4 | 删除好友解耦 | 删除好友不再影响房间生命周期 | 删除好友仅调 `removeContact`，不再 `mark/hide/leave` | 删好友后历史聊天室仍保留 | 0.5 天 |
| 5 | 详情页 UI 压缩 | 紧凑且可滚动 | 资料项单行化、间距收缩、滚动稳定 | 小屏可滚到操作区且无遮挡 | 0.5 天 |
| 6 | 聊天能力统一（关键） | 两人房可扩展为多人房 | 去掉“仅群聊可邀请/改名”限制，改为 Matrix 权限控制 | 两人房可邀第三人；改名按权限生效 | 2 天 |
| 7 | 列表模型统一（结构改） | 房间列表统一为“聊天室”模型 | 降低 `isGroup` 分叉，统一列表行为 | 列表展示与进入逻辑一致 | 2-3 天 |
| 8 | 文案与错误态完善 | 交互可理解 | 完善中英文文案、错误提示、空态引导 | 无缺失 key，提示清晰 | 0.5 天 |
| 9 | 回归与发版 | 发版前全链路稳定 | 全流程回归：加好友/接受/共同房/开新房/删好友 | 核心场景全部通过 | 1-2 天 |

## 3. 本期不做

1. 不做 Hub 数据库结构迁移。
2. 不做历史 `DEPRECATED` 房间的批量恢复/迁移脚本。
3. 不做后端 API 重构（`matrix_room_id` 改为非必填放到后续评估）。

## 4. 里程碑检查点（防偏）

1. 里程碑 A（完成 1-5）：联系人页流程稳定，好友与房间生命周期已解耦。
2. 里程碑 B（完成 6-7）：聊天室能力与列表模型统一。
3. 里程碑 C（完成 8-9）：文案完整、回归通过，可发版。

## 5. 当前状态

1. 阶段 2/3/4/5 已完成并已推送。
2. 建议后续顺序：阶段 6 -> 阶段 7 -> 阶段 8 -> 阶段 9。
